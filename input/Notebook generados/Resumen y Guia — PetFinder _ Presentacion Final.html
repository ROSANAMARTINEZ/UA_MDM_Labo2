PetFinder Adoption Prediction

Autores: Roxana Alberti · Sandra Sschicchi · Fernando Paganini · Baltazar Villanueva · Paula Calviello · Rosana Martinez
Problema: Clasificación multiclase — predecir velocidad de adopción de mascotas (AdoptionSpeed 0–4)
Métrica: Cohen's Kappa cuadrático · Dataset: ~15.000 mascotas, Malasia (Kaggle PetFinder 2019)
Resultado final: Kappa test 0.3951 (+0.0818 vs. baseline 0.3133)

1. ¿Qué problema resolvemos?

El objetivo del proyecto fue predecir con cuánta velocidad sería adoptada una mascota publicada en la plataforma PetFinder. La variable a predecir es AdoptionSpeed, con cinco clases: 0 (adoptada el mismo día), 1 (dentro de la primera semana), 2 (primera mes), 3 (entre 1 y 3 meses) y 4 (no adoptada en 100 días).

Elegimos optimizar el Cohen's Kappa cuadrático como métrica porque tiene en cuenta el acuerdo entre predicciones y etiquetas reales más allá del azar, y penaliza más los errores grandes (confundir clase 0 con clase 4 cuesta mucho más que confundir clase 0 con clase 1). Eso la hace más adecuada que el accuracy para problemas con clases ordenadas.

El dataset incluía datos multimodales: variables tabulares estructuradas (raza, edad, salud, cantidad de fotos), texto libre en el campo Description (mayormente en inglés y malayo), imágenes de las mascotas y metadatos pre-procesados de sentimiento y visión por computadora.

3. Lo que hicimos, paso a paso

EDA Análisis exploratorio

Empecé por explorar la distribución de AdoptionSpeed, las correlaciones entre variables y el contenido de los campos de texto y sentimiento. Construí un dashboard interactivo con Plotly Dash (5 tabs: distribuciones, correlaciones, texto y sentimiento, significación estadística, y resultados del modelo) que está desplegado en Render.com. El hallazgo más relevante del EDA fue que los animales con más fotos y descripciones más largas se adoptan sistemáticamente más rápido.

nb04 Feature Engineering progresivo + Optuna baseline

Implementé tres rondas de feature engineering sobre las 19 variables originales:

  • FE v1: HasPhoto, HasVideo, IsFree, AgeGroup, HealthScore, IsPureBreed, PhotoPerAnimal
  • FE v2: interacciones como Age_x_PhotoAmt, IsPureBreed_x_Age, HealthScore_x_Photo
  • FE v3: features de texto e imagen: sentiment_score, avg_label_score, desc_length, etc.

El problema en este notebook era que Optuna optimizaba los hiperparámetros evaluando directamente sobre el test set → el modelo aprendía a sobreajustarse a ese split específico. El gap train/test llegó a 0.2769.

nb05 FE v4/v5 + 5-fold CV + Target Encoding + Optuna

Este fue el notebook más importante. Introduje tres cambios metodológicos fundamentales:

1. Target Encoding con smoothing
Breed1 tiene 307 valores distintos y State tiene 15. Tratarlos como enteros no tiene sentido semántico (la raza "250" no es similar a la "251"). El target encoding reemplaza cada categoría por la media de AdoptionSpeed de esa categoría calculada sobre el train, con un factor de suavizado:

encoded = (count × mean_cat + 10 × mean_global) / (count + 10)

El smoothing evita que categorías con pocos ejemplos tengan encodings extremos — cuanto menos datos, más se acerca a la media global.
2. 5-fold Stratified CV con early stopping
Cambié la metodología de Optuna: en lugar de evaluar sobre el test set, cada trial se evalúa sobre 5 folds distintos del train. El test nunca influye en la búsqueda de hiperparámetros. Dentro de cada fold, uso early stopping (20 rondas sin mejora) para evitar que el modelo sobreajuste el fold de validación.
3. FE v4 — nuevas features sin data leakage
  • rescuer_n_pets: cantidad de mascotas publicadas por el mismo rescatador (solo conteo — sin usar el target)
  • age_rel_breed: edad del animal dividida por la mediana de su raza+tipo (captura "cachorro viejo para su raza")
  • NLP básico: word_count, unique_words, avg_word_len, uppercase_ratio, has_exclamation
  • FE v5 (aporte de un compañero): HasName y AdoptionFriction → +0.0067 kappa
⚠️ Error que detectamos y corregimos: Data Leakage
La primera versión del FE v4 calculaba rescuer_speed_mean (media de AdoptionSpeed por rescatador) sobre todo el train y lo usaba como feature. Eso produjo CV kappa 0.7685 — irreal. El problema: al computar medias del target fuera del fold, el fold de validación "sabe" la media de AdoptionSpeed de ese rescatador antes de que el modelo la aprenda. Se detecta porque CV >> test. La solución fue usar solo el conteo (rescuer_n_pets), sin involucrar el target.

nb06 Transfer Learning — DistilBERT

Hice fine-tuning de DistilBERT sobre el campo Description usando Google Colab con GPU T4 (aproximadamente 2 horas de entrenamiento). DistilBERT es una versión comprimida de BERT que mantiene el 97% de su capacidad con 40% menos de parámetros. El modelo pre-entrenado en inglés se adaptó al dominio de PetFinder con las descripciones de las mascotas.

El resultado fue que BERT solo no mejoró al modelo tabular (kappa 0.3699 vs. 0.3867). El texto ya estaba siendo capturado parcialmente por las features NLP y de sentimiento. Sin embargo, en blend con un peso pequeño (3%) sí aportó marginalmente en nb09 (+0.0029).

nb07 Selección de features: Boruta + SHAP

Apliqué dos técnicas complementarias para entender qué features importan realmente:

Boruta: crea "shadow features" (copias aleatorizadas de cada variable) y compara la importancia real contra esas sombras en 50 iteraciones con Random Forest. Una feature se acepta si supera consistentemente a sus sombras. Resultado: solo aceptó avg_label_score (1 feature), age_rel_breed quedó tentativa, y las 46 restantes fueron rechazadas.

Esto no significa que las demás sean inútiles — Random Forest no aprovecha las features tan bien como LightGBM. Boruta con RF es muy conservador para este dataset.
SHAP (SHapley Additive exPlanations): mide cuánto cambia cada predicción individual cuando se incluye o excluye cada feature, usando la teoría de juegos cooperativos (Shapley values). A diferencia del feature importance tradicional de LightGBM (que cuenta frecuencia de splits), SHAP mide el impacto real en la predicción.

Top 3 por importancia SHAP: rescuer_n_pets (0.278), age_rel_breed (0.243), Breed1_enc (0.169). Las dos features nuevas de FE v4 son las más importantes del modelo.

nb08 Ensemble LightGBM + XGBoost

Combinar LightGBM y XGBoost tiene sentido aunque ambos sean gradient boosting, porque tienen distintas estrategias de crecimiento: LightGBM crece leaf-wise (divide la hoja de mayor ganancia) y XGBoost crece depth-wise (divide todos los nodos del mismo nivel). Esas diferencias hacen que cometan errores en casos distintos, y al promediar sus probabilidades el error total se reduce.

El blend 50/50 dio kappa 0.3906. También corrí Optuna para optimizar los parámetros de XGBoost (50 trials).

nb09 Blend triple LGB + XGB + BERT

Sumé las predicciones de DistilBERT al blend tabular. Con un grid search de pesos encontré que la combinación óptima es LGB 53.4% + XGB 43.7% + BERT 3%. BERT con poco peso aporta información semántica que los otros dos no capturan directamente desde las features tabulares. Resultado: kappa 0.3935.

nb10 CatBoost + ensemble triple tabular

Agregué CatBoost como tercer gradient boosting tabular. Su diferencial técnico es el crecimiento depth-wise simétrico: todos los nodos del mismo nivel se dividen con la misma condición, lo que lo hace más conservador que LGB y menos propenso al overfitting. También tiene manejo nativo de variables categóricas via ordered target encoding interno.

Individualmente CatBoost obtuvo kappa 0.3489 (por debajo de LGB y XGB), pero al combinarlo en el ensemble aportó diversidad suficiente para llegar al nuevo record: LGB 30% + XGB 45% + CB 25% = 0.3951.

4. Conceptos clave para explicar en la presentación

¿Por qué 5-fold CV en lugar de train/test simple?

Con un único split, Optuna puede encontrar parámetros que funcionan bien específicamente en ese test. Al evaluarlos siempre sobre los mismos datos, los hiperparámetros se sobreajustan a ese split particular. Con 5-fold CV, cada combinación de parámetros debe funcionar bien en cinco contextos distintos, lo que garantiza mayor generalización. El test set nunca se usa durante la búsqueda.

¿Por qué Target Encoding y no One-Hot Encoding?

Breed1 tiene 307 categorías. One-Hot generaría 307 columnas binarias — curse of dimensionality. Target encoding colapsa eso en una sola columna continua que captura información real del target para cada raza. El smoothing asegura que razas con pocos animales no tengan encodings extremos e inestables.

¿Por qué SHAP es mejor que el feature importance de LightGBM?

Feature importance tradicional cuenta cuántas veces se usó cada feature para hacer splits. Una feature puede aparecer muchas veces pero con impacto pequeño en la predicción, o pocas veces pero con impacto enorme. SHAP calcula la contribución marginal real de cada feature a cada predicción individual, usando el concepto de Shapley values de la teoría de juegos cooperativos. Es más costoso computacionalmente pero mucho más informativo e interpretable.

¿Por qué funciona el ensemble si los modelos son similares?

LGB, XGB y CatBoost comparten la misma familia (gradient boosting) pero tienen sesgos inductivos distintos. LGB tiende a overfit en hojas muy específicas (leaf-wise). XGB produce árboles más balanceados (depth-wise). CatBoost produce árboles completamente simétricos. Como cada uno sobreajusta en casos distintos, al promediar sus probabilidades los errores individuales se cancelan. La clave es la diversidad de los errores, no la similitud de los modelos.

¿Por qué BERT con solo 3% de peso?

BERT captura información semántica del texto crudo, pero gran parte de esa información ya está representada en las features NLP básicas que tenemos (word_count, sentiment_score, avg_label_score, etc.). El campo Description en PetFinder además es corto y ruidoso — mezcla inglés y malayo con muchos errores de tipeo. Con más de un 5-10% de peso, BERT introduce ruido que supera su aporte. Al 3% captura solo la información marginal que los modelos tabulares no ven directamente.

¿Qué aprendimos del data leakage?

Cuando calculamos la media de AdoptionSpeed por rescatador sobre todo el train y la usamos como feature, el modelo podía "ver" indirectamente el target de los animales del fold de validación. El CV kappa llegó a 0.7685 — claramente irreal comparado con el 0.45 de test. La señal de alerta fue exactamente esa: una brecha enorme entre CV y test. La lección es que cualquier feature que se construya usando el target debe calcularse exclusivamente dentro del fold de train, nunca sobre datos que el modelo vea como validación.

¿Por qué Boruta rechaza casi todo si las features son útiles?

Boruta usa Random Forest internamente, y RF no aprovecha las features tan eficientemente como LightGBM. RF no usa early stopping, no hace target encoding, y selecciona features al azar en cada nodo. El umbral estadístico de Boruta (95% de confianza en 50 iteraciones) es muy conservador. El resultado — solo 1 feature aceptada — no refleja que las demás sean inútiles, sino que RF no puede distinguirlas del ruido con la potencia estadística del test. SHAP con LightGBM confirma que las 48 features tienen importancia real.
